# Openclaw → Jiner 自然语言自动路由 - 技术设计

## Context

### 背景

现有 privy-jiner 插件 (`~/.openclaw/extensions/privy-jiner/`) 通过显式工具调用 `jiner_nlu` 工作：

```
用户: "调用 jiner_nLU 工具处理: 我喝了30ml水"
```

这种方式要求用户了解系统能力边界，不够自然。

### 当前架构

```
┌─────────────────────────────────────────────────┐
│           Openclaw Gateway                      │
│  (消息入口: QQ/Telegram/Slack/Discord)         │
└─────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│         privy-jiner 插件                        │
│  - tools: jiner_nlu (显式调用)                 │
│  - capabilities: health_log_water, finance_*   │
└─────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│           privy-jiner API 服务                   │
│  - /api/ai/nlu (自然语言理解)                   │
│  - /api/health/*, /api/finance/*               │
└─────────────────────────────────────────────────┘
```

### 约束

1. **现有代码**: privy-jiner 插件已实现 NLU 端点 (`/api/ai/nlu`)
2. **Openclaw Gateway**: 需要确认是否支持插件扩展路由逻辑
3. **最小改动**: 尽量复用现有能力，不做大幅重构

## Goals / Non-Goals

### Goals

1. 实现自然语言输入的**自动意图识别**
2. 用户无需显式调用工具，系统自动路由到 jiner 处理
3. 配置化意图规则，支持灵活扩展
4. 保持现有功能完整，平滑迁移

### Non-Goals

- 不修改 Openclaw Gateway 核心代码（如果支持插件扩展则利用扩展点）
- 不支持多语言意图识别（第一版仅中文）
- 不实现复杂的 ML 意图分类（使用关键字 + 轻量语义匹配）

## Decisions

### D1: 路由实现位置 - "插件层实现"

**选择**: 在 privy-jiner 插件中实现意图路由，通过 Gateway 扩展点注册

**理由**:
- 插件已具备 NLU 端点，可以复用
- Gateway 扩展机制支持 Tool 注册，可拦截消息
- 保持职责清晰：jiner 插件负责自己的意图识别

**备选方案**:
| 方案 | 优点 | 缺点 |
|------|------|------|
| Gateway 核心实现 | 统一路由 | 需要修改核心代码 |
| 插件层实现（选用）| 复用现有代码，解耦 | 需要 Gateway 扩展点支持 |
| 独立 Router 服务 | 灵活扩展 | 增加部署复杂度 |

### D2: 意图识别策略 - "双层匹配"

**选择**: 关键字匹配 + NLU API 分类

```
用户消息 → 关键字预检 → NLU 分类 → 路由决策
     │              │
     │              └── 快速判断意图类型
     │
     └── 直接匹配已知意图
```

**理由**:
- 关键字匹配：高效，处理常见场景（"喝水"、"吃饭"、"花了"）
- NLU 分类：处理复杂、模糊的自然语言
- 两层保障，减少误判

**备选方案**:
| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯关键字 | 简单快速 | 覆盖度低 |
| 纯 NLU API | 覆盖度高 | 延迟增加，费用 |
| 双层匹配（选用）| 平衡效率与准确率 | 实现复杂度稍高 |

### D3: 配置化路由规则

**选择**: 在 `openclaw.json` 中配置意图映射规则

```json
{
  "plugins": {
    "entries": {
      "privy-jiner": {
        "enabled": true,
        "intentRules": {
          "keywords": {
            "health": ["喝水", "运动", "跑步", "锻炼"],
            "finance": ["花钱", "买了", "支付", "工资", "收入"],
            "fashion": ["衣服", "穿搭", "衣柜", "买衣服"]
          },
          "fallbackToNlu": true,
          "nluThreshold": 0.7
        }
      }
    }
  }
}
```

### D4: 意图分类 API 设计

在 privy-jiner 插件中新增意图分类端点：

```
POST /api/ai/classify-intent
{
  "message": "我喝了30ml水"
}

Response:
{
  "intent": "health_log_water",
  "confidence": 0.95,
  "entities": {
    "amount": 30,
    "unit": "ml"
  }
}
```

## Risks / Trade-offs

### R1: Gateway 扩展点限制

**风险**: Openclaw Gateway 可能不支持插件自定义消息拦截

**缓解**:
- 方案A: 使用现有 Tool 机制，声明支持"自动调用"模式
- 方案B: 通过 Webhook 机制实现消息拦截
- 需要实际测试 Gateway 扩展能力

### R2: 意图识别准确率

**风险**: 关键字匹配可能误判，NLU API 可能返回低置信度

**缓解**:
- 配置 `confidence threshold`，低于阈值时询问用户
- 提供"否认"机制，用户可以纠正错误意图
- 持续优化关键字词库

### R3: 性能延迟

**风险**: 引入 NLU 分类增加响应延迟

**缓解**:
- 关键字匹配优先，快速路径
- NLU 调用异步化（可选）
- 缓存常见意图结果

### R4: 配置复杂度

**风险**: 意图规则配置可能变得复杂难维护

**缓解**:
- 提供默认配置，开箱即用
- 配置文档化
- 考虑提供 UI 界面管理规则

## Migration Plan

### Phase 1: 插件扩展
1. 修改 `openclaw.plugin.json`，新增 `nlu_handoff` capability
2. 实现 `/api/ai/classify-intent` 端点
3. 测试 NLU 分类功能

### Phase 2: Gateway 集成
1. 研究 Openclaw Gateway 扩展机制
2. 实现意图路由拦截器
3. 配置关键字规则

### Phase 3: 测试与优化
1. 端到端测试自然语言输入
2. 优化意图识别准确率
3. 文档完善

### Rollback 策略

- 如果 Gateway 扩展不可行：回退到显式工具调用模式
- 配置化设计支持快速关闭自动路由
